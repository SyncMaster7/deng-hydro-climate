# deng-hydro-climate

A production-like data engineering pipeline monitoring Estonian river and climate conditions, built on a self-hosted server.

## Business Question

> How do weather conditions — precipitation, temperature, and snow depth — drive water levels and runoff across Estonian river catchments, and which rivers respond most strongly and most quickly to rainfall events?

## Overview

This pipeline ingests hourly hydrological and meteorological measurements from the Estonian Environment Agency APIs, stores raw responses in a landing zone, ingests them into a bronze layer, and transforms them through a medallion architecture (bronze → silver → gold) using dbt. Station proximity is auto-calculated using the haversine formula — no manually maintained CSV.

## Data Sources

| Source | Endpoint | Description | Stations | Granularity |
|---|---|---|---|---|
| Hydrological API | `keskkonnaandmed.envir.ee/f_hydroseire` | Water level, temperature, runoff | 76 hydrometric stations | Hourly (HLY) |
| Meteorological API | `keskkonnaandmed.envir.ee/f_kliima_tund` | Precipitation, temperature, snow depth, wind | 25 meteorological stations | Hourly (HLY) |
| Hydrometric stations | `seeds/hydrometric_stations.csv` | 76 stations with full metadata incl. gauge zero elevation | Reference data | Static |
| Meteorological stations | `seeds/meteorological_stations.csv` | 25 stations with full metadata | Reference data | Static |
| Station proximity | Auto-generated by seed DAG (haversine) | Top 3 nearest meteo stations per hydro station | 228 pairs | Regenerated on station change |

## Pipeline Architecture

### Medallion Layers

| Layer | Location | Contents |
|---|---|---|
| Raw | `/data/raw/{source}/` | API responses as `.json` files — exact, unmodified |
| Bronze | `bronze.hydro`, `bronze.meteo` | Ingested from raw files — no filtering, no transformation |
| Silver | dbt models | Cleaned, typed, pivoted from long to wide format |
| Gold | dbt models | Aggregated, analysis-ready, coastal offset applied |

### Data Flow

```
Seed DAG (manual):
  hydrometric_stations.csv ──► load_hydrometric_stations ──► public.hydrometric_stations ──► asset
  meteorological_stations.csv ─► load_meteorological_stations ─► public.meteorological_stations ─► asset
                                              ↓ (if changed or proximity empty)
                                    calculate_proximity (haversine)
                                              ↓
                                    public.station_proximity ──► asset

hydro_meteo_pipeline DAG (daily):
  Hydro API ──► fetch_hydro ──► /data/raw/hydro/hydro_{date}.json ──► asset:hydro_raw_file
  Meteo API ──► fetch_meteo ──► /data/raw/meteo/meteo_{date}.json ──► asset:meteo_raw_file
                                              ↓ (asset-driven)
                         ingest_hydro ──► bronze.hydro ──► asset:hydro_bronze
                         ingest_meteo ──► bronze.meteo ──► asset:meteo_bronze
                                              ↓ (asset-driven, both bronze assets ready)
                                           run_dbt
                                    (bronze → silver → gold)

archive_raw_files DAG (daily, independent):
  /data/raw/ ──► files older than 7 days ──► gzip ──► /data/archive/{source}/{year}/{month}/
```

## Database Schema

### bronze.hydro
Ingested from `/data/raw/hydro/hydro_{date}.json`. No filtering applied.

`UNIQUE (jaam_kood, timeline_ts_utc, aegrida_nimi)`

Fields: `id`, `jaam_kood`, `jaam_nimi`, `valgala_nimi`, `valgala_suurus_km2`, `kaugus_suudmest_km`, `jaam_laiuskraad`, `jaam_pikkuskraad`, `veekogu_nimi`, `timeline_ts_utc`, `timeline_ts_local`, `aegrida_nimi`, `vaartus`, `loaded_at`

### bronze.meteo
Ingested from `/data/raw/meteo/meteo_{date}.json`. No filtering applied.

`UNIQUE (jaam_kood, aasta, kuu, paev, tund, element_kood)`

Fields: `id`, `jaam_kood`, `jaam_nimi`, `aasta`, `kuu`, `paev`, `tund`, `vaartus`, `element_kood`, `element_nimi`, `element_yhik`, `observation_ts`, `loaded_at`

### bronze.etl_log
Fields: `id`, `dag_id`, `task_id`, `run_date`, `started_at`, `finished_at`, `rows_loaded`, `source_file`, `status`, `error_message`

### public.hydrometric_stations
`UNIQUE (station_code)` — INTEGER

Fields: `id`, `station_code`, `station_category`, `station_name`, `station_fullname`, `water_body`, `catchment_name`, `catchment_size_km2`, `distance_from_mouth_km`, `station_altitude_msl_m`, `latitude`, `longitude`, `is_active`, `loaded_at`

### public.meteorological_stations
`UNIQUE (station_code)` — TEXT

Fields: `id`, `station_code`, `station_category`, `station_name`, `latitude`, `longitude`, `altitude_m`, `is_active`, `loaded_at`

### public.station_proximity
Auto-generated by seed DAG — not loaded from CSV.

`UNIQUE (hydro_station_code, proximity_rank)`

Fields: `id`, `hydro_station_code`, `meteo_station_code`, `distance_km`, `proximity_rank` (1–3), `loaded_at`

> SCD Type 2 on all three station tables is implemented via dbt snapshots.

## Getting Started

### Prerequisites
- Docker and Docker Compose installed on `deng-vm`
- SSH access configured (`~/.ssh/config` with `deng-vm` shortcut)

### Setup

1. Clone the repository:
   ```bash
   git clone git@github.com:SyncMaster7/deng-hydro-climate.git
   cd deng-hydro-climate
   ```

2. Copy the environment file and fill in credentials:
   ```bash
   cp .env.example .env
   # Edit .env — never commit this file
   ```

3. Create host data directories:
   ```bash
   sudo mkdir -p /data/raw/hydro /data/raw/meteo
   sudo mkdir -p /data/archive/hydro /data/archive/meteo
   sudo mkdir -p /data/volumes/postgres_data /data/volumes/airflow_db
   sudo chown -R 999:999 /data/volumes/postgres_data /data/volumes/airflow_db
   ```

4. Start services:
   ```bash
   docker compose up -d
   ```

5. Trigger the **seed_stations** DAG manually from the Airflow UI to populate station tables and auto-generate the proximity matrix.

6. Trigger **hydro_meteo_pipeline** for a single day to verify the full fetch → ingest → dbt flow.

7. Run a backfill once the pipeline is verified:
   ```bash
   airflow backfill create --max-active-runs 1 --run-backwards
   ```

## Known Issues

- **PostgreSQL auth:** Both DB services use `POSTGRES_HOST_AUTH_METHOD: md5` — required for psycopg2 compatibility with Airflow. Volume must be wiped for this to take effect after a change.
- **Airflow 3 JWT:** `AIRFLOW__API_AUTH__JWT_SECRET` must be set identically across all Airflow services. Without it, LocalExecutor tasks fail with `Signature verification failed`.
- **Windows BOM:** CSV files created on Windows may have a UTF-8 BOM. Use `encoding='utf-8-sig'` when reading CSVs in Python.
- **Coastal water level datum:** All 15 coastal hydrometric stations have gauge zero at −500 cm EH2000. Raw readings are not directly comparable to river stations — the offset is applied in dbt using `station_altitude_msl_m`.
- **Volume permissions:** After wiping a volume, run `sudo chown -R 999:999 /data/volumes/postgres_data` and `sudo chown -R 999:999 /data/volumes/airflow_db` before restarting services.
